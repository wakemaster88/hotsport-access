Neuen Raspberry Pi einrichten

Diese Anleitung führt dich in fünf Schritten durch die Inbetriebnahme eines neuen Drehkreuz-Pis. Geplante Dauer: ~10 Minuten pro Pi (zzgl. Zeit für das Image-Flashen).

Material & Voraussetzungen

  • Raspberry Pi (3B+, 4 oder Zero 2 W) mit microSD-Karte (min. 8 GB)
  • Netzteil (offizielles Pi-Netzteil empfohlen) und LAN- oder WLAN-Verbindung ins selbe Netz wie das Drehkreuz (für Binarytec-API)
  • Scanner-Hardware: USB-QR-/RFID-Wedge-Scanner, USB-Kamera oder MFRC522-RFID-Modul (siehe Schritt 5)
  • Relais-Modul (5 V, aktiv-high) am GPIO 24 und Buzzer am GPIO 23 (Standard, in devices.json pro Pi änderbar)
  • Den Binarytec-API-Bearer-Token (Pflicht – ohne den kann der Pi keine Scans validieren).
  • Optional: den Hub-Pi-Token für die Dashboard-Anzeige – findest du oben in dieser Anleitung. Ohne Hub-Token läuft der Pi im Standalone-Modus (kein Heartbeat, Scans funktionieren trotzdem).
  • Einen Eintrag in der zentralen devices.json (siehe Schritt 2).
  • Internet auf dem Pi – nur beim Bootstrap. Der Installer zieht ~30 MB Pakete (apt + pip + git clone). Nach Schritt 3 kann der Pi dauerhaft offline (vom Internet) bleiben – er braucht nur noch das LAN zur Binarytec-API.
1

Raspberry Pi OS Lite flashen

Lade den Raspberry Pi Imager und flashe Raspberry Pi OS Lite (64-bit) auf die SD-Karte. Im Imager unter „OS-Anpassungen" bitte folgendes setzen:

  • Hostname: hotsport-pi-NN (z.B. hotsport-pi-01)
  • SSH aktivieren – mit Passwort oder Public-Key
  • Benutzer: pi oder ein eigener Admin-User
  • WLAN/Ethernet-Konfig falls nicht per LAN angeschlossen
  • Locale & Tastatur: Deutsch

Tipp: Nimm für jeden Pi einen eindeutigen Hostnamen. Der Hub nutzt diesen als Default-pi_id, falls du keine eigene setzt.

2

Pi in devices.json eintragen

Alle Drehkreuz-Pis stehen zentral in pi-app/devices.json im Repo. Diese Datei enthält:

  • hub.base_url – LAN-URL dieses Hubs (aktuell: {{ hub_url }}), einmal global gesetzt.
  • api.base_url + verify_tls – Binarytec-Endpoint, einmal global.
  • defaults – Standard-GPIO-Pins, Reader-Pfade etc.
  • pis[] – pro Pi: pi_id, name, location, reader_mode, inout, interface_id (und optional individuelle Pins).

Keine Tokens in devices.json – die werden auf dem Pi interaktiv eingegeben (Schritt 3). 

# Auf deinem Mac/PC im geklonten Repo:
$EDITOR pi-app/devices.json
git add pi-app/devices.json
git commit -m "add hotsport-pi-04"
git push
{
  "pi_id":        "hotsport-pi-04",
  "name":         "Eingang West",
  "location":     "hotsport",
  "reader_mode":  "keyboard",
  "inout":        "in",
  "interface_id": "104"
}

reader_mode: keyboard (USB-Wedge, Standard), qr_camera oder rfid_mfrc522. Pins/Devicepfade abweichend? Einfach im selben Objekt relay_pin, buzzer_pin, reader_device_path oder reader_camera_index ergänzen – sonst gelten die defaults.

3

Pi-App auf dem Pi installieren

SSH auf den Pi, Repo klonen, Installer aufrufen. Der fragt interaktiv nach Pi-Auswahl, API-Token und (optional) Hub-Token – generiert daraus die komplette /etc/hotsport-access/config.toml.

Wichtig: Der Pi braucht für diesen einen Schritt Internet (apt + pip + git, ~30 MB). Danach kann er vom Internet getrennt werden – Pi und Drehkreuz-API müssen nur im selben LAN sein. 

sudo apt-get update
sudo apt-get install -y git
git clone https://github.com/wakemaster88/hotsport-access.git
cd hotsport-access/pi-app
sudo bash scripts/install.sh

Der Installer zeigt die Liste aus devices.json:

Verfügbare Pis aus devices.json:

   1) hotsport-pi-01           Eingang Nord          hotsport      keyboard      in
   2) hotsport-pi-02           Eingang Süd           hotsport      keyboard      in
   3) hotsport-pi-03           Ausgang Nord          hotsport      keyboard      out

Auswahl (Nummer 1-3): _

Danach kommen zwei Tokens ins Spiel. Reihenfolge der Quellen (erste Übereinstimmung gewinnt):

  1. Argumente beim Aufruf: sudo bash install.sh -y hotsport-pi-01 API_TOKEN HUB_TOKEN
  2. Env-Variablen: HOTSPORT_API_TOKEN=… HOTSPORT_HUB_TOKEN=… sudo -E bash install.sh -y …
  3. Vorhandene /etc/hotsport-access/config.toml – Re-Install läuft komplett automatisch, weil die Tokens vom ersten Lauf wiederverwendet werden.
  4. devices.json: api.bearer_token / hub.pi_token. Achtung Tokens im Repo bedeuten: Repo MUSS privat sein.
  5. Interaktiv (verdeckte Eingabe am Pi). Diese Eingabe muss nur einmal pro Pi passieren – danach läuft Punkt 3.

Binarytec-API-Bearer-Token ist Pflicht (ohne ihn keine Ticket-Validierung). Hub-Pi-Token ist optional – leer = Standalone-Modus (Pi scant ohne Hub, taucht aber nicht im Dashboard auf). {% if pi_token %} Hub-Token findest du oben in dieser Anleitung. {% else %} Aktuell kein Hub-Token gesetzt – kein Problem, Pi läuft trotzdem (Standalone). {% endif %}

Direkter Aufruf mit Pi-ID + Tokens (überspringt alle Prompts) 
# Mit Hub:
sudo bash scripts/install.sh -y hotsport-pi-04 BINARYTEC-TOKEN HUB-TOKEN

# Standalone (ohne Hub):
sudo bash scripts/install.sh -y hotsport-pi-04 BINARYTEC-TOKEN ""

Idempotent – beim zweiten Lauf wird die alte config.toml als .bak.timestamp gesichert und neu geschrieben. venv und Daten bleiben erhalten.

Bei aktiviertem Hub-Token taucht der Pi innerhalb ~5 Sekunden unten im Dashboard auf. Routine-Updates verteilt der hotsport-updater-Service automatisch über den Hub.

4

Optional: im Dashboard nachjustieren

Alle pi-spezifischen Werte (Reader-Modus, Richtung, Interface-ID, GPIO-Pins, …) kommen primär aus devices.json. Damit ist kein Dashboard-Eingriff für den Standard-Betrieb nötig.

Wer einen Pi nur für diesen Pi kurzfristig anders konfigurieren will (z.B. Pin-Test, Disable), kann das über das Hub-Dashboard tun – das überschreibt die devices.json-Werte via Live-Config-Push:

Interface-ID
Die resourceId aus der Binarytec-API (z.B. 101).
Richtung
in (Eingang) oder out (Ausgang).
Reader-Devicepfad / Kamera-Index
Default "auto" – der Pi sucht beim Start automatisch ein USB-Keyboard mit Buchstaben/Ziffern (= Barcode-Scanner) unter /dev/input/event* und filtert HDMI-Inputs raus. Nur überschreiben, wenn mehrere passende Geräte angeschlossen sind: dann z.B. /dev/input/by-id/usb-…-event-kbd. Bei Kameras meist 0.
GPIO-Pins
Standard: Relais GPIO 24, Buzzer GPIO 23, Pulsdauer 1.0 s. Nur ändern, wenn die Verkabelung abweicht.

Voraussetzung: der Pi wurde mit Hub-Token installiert. Standalone-Pis (ohne Hub-Token) sehen Dashboard-Änderungen nicht – dort dauerhafte Anpassungen bitte in devices.json committen und neu installieren.

5

Reader-spezifische Hinweise

--mode beim Bootstrap entscheidet, welche Pakete der Installer mit-installiert. Wenn du den Modus später wechseln willst, reicht meistens das Setzen im Dashboard plus das Nachinstallieren der Reader-Pakete:

USB-Kamera (qr_camera): 

sudo apt-get install -y python3-opencv
sudo /opt/hotsport-access/venv/bin/pip install \
    -r /opt/hotsport-access/current/requirements-camera.txt
sudo systemctl restart hotsport-access

MFRC522-RFID (SPI): Einmalig sudo raspi-config → Interface Options → SPI aktivieren, dann: 

sudo apt-get install -y python3-spidev
sudo /opt/hotsport-access/venv/bin/pip install \
    -r /opt/hotsport-access/current/requirements-rfid.txt
sudo systemctl restart hotsport-access
6

Health & Logs am Pi prüfen

Wenn ein Pi sich seltsam verhält oder du ihn nach der Installation verifizieren willst, sind diese Befehle deine erste Anlaufstelle. Alle laufen direkt auf dem Pi (per ssh pi@<pi-ip>).

Service-Status

Zeigt, ob der Dienst läuft, wann er zuletzt neu gestartet hat und die ersten Log-Zeilen seit dem Start. 

sudo systemctl status hotsport-access --no-pager
sudo systemctl status hotsport-updater --no-pager

Health-Endpoint (lokaler Selbsttest)

Der Pi-Daemon antwortet lokal auf Port 8765 mit Healthy-Flag, Versions- und Reader-Info – ideal um zu sehen, ob er Konfig hat. 

curl -s http://127.0.0.1:8765/health | python3 -m json.tool

Live-Logs

-f bleibt offen und zeigt neue Zeilen sofort an (Strg+C zum Beenden). Sehr nützlich beim Scannen, um Reader-Events und API-Antworten in Echtzeit zu sehen. 

journalctl -u hotsport-access -f --no-hostname

Letzte 100 Log-Zeilen

Schneller Rückblick ohne Live-Folgen – gut um nach einem Vorfall zu schauen, was zuletzt passiert ist. 

journalctl -u hotsport-access -n 100 --no-pager

Nur Fehler / Warnungen

Filtert die Logs auf WARNING und höher – bei langer Uptime der schnellste Weg zu echten Problemen. 

journalctl -u hotsport-access -p warning --since "1 hour ago" --no-pager

Lokales Ereignis-Log (letzte 100 Scans & Service-Events)

Pi speichert die letzten 100 Ereignisse in /var/lib/hotsport-access/state.sqlite3 – dieselben, die im Hub-Dashboard pro Pi-Detail erscheinen, plus die offline angesammelten. 

sudo sqlite3 /var/lib/hotsport-access/state.sqlite3 \
  "SELECT datetime(scanned_at,'unixepoch','localtime'), kind, code, granted, reason, pushed
   FROM scans ORDER BY scanned_at DESC LIMIT 100;"

Service neustarten / stoppen

Nach manuellen Änderungen an config.toml oder dem Code. Der hotsport-updater startet den Daemon automatisch wieder, falls er crasht. 

sudo systemctl restart hotsport-access
sudo systemctl stop hotsport-access     # zum Anhalten (Updater bleibt aus)
sudo systemctl start hotsport-access

Konfiguration sehen (ohne Token)

Zeigt die aktive config.toml ohne Bearer-/Pi-Token, damit man sie gefahrlos kopieren oder posten kann. 

sudo grep -vE '^(bearer_token|pi_token)' /etc/hotsport-access/config.toml

Troubleshooting

Pi taucht nicht im Dashboard auf
  • Wurde der Pi mit Hub-Token installiert? Ohne ist Standalone-Modus aktiv (kein Dashboard-Eintrag). Prüfen: grep pi_token /etc/hotsport-access/config.toml – wenn die Zeile fehlt: Pi neu installieren mit Hub-Token.
  • Auf dem Pi: sudo systemctl status hotsport-access – Dienst aktiv?
  • Logs: journalctl -u hotsport-access -n 50 --no-pager
  • Vom Pi aus den Hub testen: curl -i {{ hub_url }}/health
  • Token korrekt? In /etc/hotsport-access/config.toml muss [hub] pi_token exakt dem HOTSPORT_HUB_PI_TOKEN auf dem Hub entsprechen.
Pi scant nicht (Standalone funktioniert auch nicht)
  • API-Token gesetzt? grep bearer_token /etc/hotsport-access/config.toml
  • Drehkreuz-API erreichbar? curl -kI https://192.168.251.50:444/ (URL aus devices.json)
  • Logs zeigen API-Fehler? Token oder interface_id kontrollieren – beides muss zur Binarytec-Konfiguration passen.
  • Health-Endpoint: curl -s http://127.0.0.1:8765/health | python3 -m json.tool
Bootstrap bricht ab beim apt-get install

Der Pi braucht für Schritt 2 Internet. Test: ping -c 2 8.8.8.8 und curl -I https://github.com. Falls beides fehlschlägt: Pi temporär ans normale WLAN hängen (sudo raspi-config → System Options → Wireless LAN), Bootstrap durchlaufen lassen, danach zurück ins Hub-Netz.

Code-Update auf einem bereits installierten Pi

Im Normalfall kommen Updates vom Hub. Falls du den Pi-App-Code manuell aus GitHub aktualisieren willst (z.B. nach main-Branch-Änderungen): 

cd ~/hotsport-access && git pull
cd pi-app && sudo bash scripts/install.sh keyboard
sudo systemctl restart hotsport-access

config.toml, venv und Daten bleiben erhalten – der Installer ist idempotent.

Status bleibt „unkonfiguriert"

Sollte mit der neuen devices.json-basierten Installation nicht mehr passieren. Wenn doch:

  • Auf dem Pi: cat /etc/hotsport-access/config.toml – sind die Sections [api], [pi] und [pi.reader] vorhanden?
  • Sind api.base_url, api.bearer_token und pi.interface_id alle nicht-leer?
  • Logs prüfen: journalctl -u hotsport-access -n 100 --no-pager
  • Im Zweifel: install.sh nochmal laufen lassen – sichert die alte config.toml als .bak.*.
Reader liest nichts ein
  • Wedge-Scanner: ls /dev/input/by-id/ – den passenden Pfad als Reader-Devicepfad ins Dashboard eintragen.
  • Kamera: v4l2-ctl --list-devices auf dem Pi installieren und richtigen Index setzen.
  • RFID: SPI in raspi-config aktiv? User hotsport ist Mitglied der Gruppe spi?
Drehkreuz öffnet nicht

Im Dashboard prüfen: ist der Pi aktiv? Stimmt der Relais-GPIO? Test direkt auf dem Pi: 

sudo /opt/hotsport-access/venv/bin/python -c \
  "from gpiozero import OutputDevice; import time; \
   r = OutputDevice(24); r.on(); time.sleep(1); r.off()"
